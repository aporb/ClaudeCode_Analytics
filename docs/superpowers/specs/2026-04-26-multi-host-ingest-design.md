## 1. Goal

Capture Claude Code transcript data from two SSH-accessible remote machines — `ssh_hostinger` and `ssh_picoclaw` — into the same local Postgres that already holds local-Mac data, tag every row with the originating host, and surface the resulting per-host breakdowns in the existing web UI. Make total tokens (in / out / cache / total) a first-class headline metric on the home page.

The system is currently single-machine by design: a `chokidar` tailer in `apps/ingester/src/daemon/` watches `~/.claude` on this Mac and writes through `@cca/db` to Postgres in a Supabase container. Nothing in the pipeline knows about more than one machine. This spec adds that without disturbing the existing live-capture path.

---

## 2. Constraints & non-goals

**Constraints:**
- No remote installs. The remotes already run Claude Code; we do not want to install Node, this repo, or any agent on them.
- No exposed Postgres. The DB stays bound to `localhost:54322`.
- The user's `~/.ssh/config` already defines `ssh_hostinger` and `ssh_picoclaw` as working aliases.
- macOS-host-only. Same FDA / launchd caveats as the existing system.
- Single user, single DB. No auth, no multi-tenant concerns.

**Non-goals:**
- No live SSE for remote events. Remotes are batched-pull only; the live indicator stays meaningful for *local* activity.
- No remote daemons or hooks. Remote sessions never receive `status='active'` (only the local hook-relay path sets that).
- No bidirectional sync; pull-only.
- No SSH key management; relies on existing config.
- No cross-host session-id collision handling. Theoretically impossible (UUIDs); not engineered for.
- No alerting beyond an in-UI banner when sync is repeatedly failing.

## 3. Architecture

```
┌──────────────────────────────────────────────────────────────────┐
│  Mac ("local")                                                   │
│                                                                  │
│   ~/.claude    ──►  daemon  ──►  ┌────────────────────┐          │
│                  (live tail,     │  Postgres :54322   │          │
│                   host='local')  │                    │          │
│                                  │  + host columns    │          │
│   <repo>/.cca/remotes/           │  + host_sync_state │          │
│     ├─ hostinger/.claude/  ──►   │  + cca.remotes.json│          │
│     └─ picoclaw/.claude/   ──►   └─────────┬──────────┘          │
│            (rsync mirrors)                 │                     │
│                                            ▼                     │
│   launchd every 3h ──► `cca sync`        Web UI :3939            │
│       • reads cca.remotes.json           + /hosts page           │
│       • per-host backoff state           + host filter chip      │
│       • SSH+rsync each due host          + token headline on /   │
│       • re-uses backfill orchestrator    + host badge on detail  │
└──────────────────────────────────────────────────────────────────┘
```

The system gains exactly one new long-running scheduled job (a launchd agent that runs `cca sync` every 3 hours) and one new persistent state row per host. The daemon is unchanged. The parsers are unchanged. The writers gain a single `host` parameter that defaults to `'local'` so the existing local path is unaffected.

The five new pieces:

1. **`cca.remotes.json`** at the repo root — the user-editable registry of remotes to pull from.
2. **`apps/ingester/src/sync/`** — new module: rsync runner, per-host orchestrator, backoff state machine. Surfaced as `cca-ingester sync` and a wrapper `cca sync`.
3. **`host_sync_state` table** — one row per host, mutated by the sync runner.
4. **Schema additions** — `host TEXT NOT NULL DEFAULT 'local'` on every event-derived table, two new indexes on `(host, timestamp)` and `(host, started_at)`, and a `usage_daily` materialized-view rebuild that adds `host` to its grouping.
5. **Web UI additions** — host filter chip in nav, new `/hosts` page, token headline on `/`, host badges on session detail.

## 4. Schema

### 4.1 Column additions and `usage_daily` rebuild

A single hand-authored SQL supplement `packages/db/drizzle/0011_multi_host.sql`. **Not generated by drizzle-kit.** The drizzle journal at `packages/db/drizzle/meta/_journal.json` only registers entries through `0003_ambiguous_jean_grey`; every higher-numbered file in this repo (`0001_events_gin.sql`, `0003_messages_indexes.sql`, `0005_prompts_history_dedup_fix.sql`, `0010_usage_daily_view.sql`) is a hand-authored supplement that `pnpm db:migrate` applies directly. `0011_multi_host.sql` follows the same convention. We do NOT regenerate the drizzle artifacts for this change; instead, we update `packages/db/src/schema/*.ts` so `pnpm db:push` against the test DB stays consistent.

```sql
-- Column additions. ADD COLUMN ... DEFAULT 'literal' is metadata-only on PG 11+;
-- existing rows automatically receive 'local' without a table rewrite.
ALTER TABLE events           ADD COLUMN host TEXT NOT NULL DEFAULT 'local';
ALTER TABLE sessions         ADD COLUMN host TEXT NOT NULL DEFAULT 'local';
ALTER TABLE messages         ADD COLUMN host TEXT NOT NULL DEFAULT 'local';
ALTER TABLE tool_calls       ADD COLUMN host TEXT NOT NULL DEFAULT 'local';
ALTER TABLE prompts_history  ADD COLUMN host TEXT NOT NULL DEFAULT 'local';
ALTER TABLE file_snapshots   ADD COLUMN host TEXT NOT NULL DEFAULT 'local';
ALTER TABLE shell_snapshots  ADD COLUMN host TEXT NOT NULL DEFAULT 'local';
ALTER TABLE todos            ADD COLUMN host TEXT NOT NULL DEFAULT 'local';

CREATE INDEX events_host_ts_idx        ON events   (host, timestamp DESC);
CREATE INDEX sessions_host_started_idx ON sessions (host, started_at DESC);

-- Rebuild usage_daily with host in the grouping. The materialized view defined
-- in 0010_usage_daily_view.sql currently groups by (day, project_path, model);
-- without host in the key, every per-host token aggregate that hits the view
-- silently sums across hosts. Rebuilding is cheap (single SELECT over messages).
DROP MATERIALIZED VIEW IF EXISTS usage_daily;
CREATE MATERIALIZED VIEW usage_daily AS
SELECT
  date_trunc('day', m.timestamp)            AS day,
  m.host                                    AS host,
  s.project_path                            AS project_path,
  m.model                                   AS model,
  COUNT(*)                                  AS message_count,
  COALESCE(SUM(m.input_tokens), 0)          AS input_tokens,
  COALESCE(SUM(m.output_tokens), 0)         AS output_tokens,
  COALESCE(SUM(m.cache_creation_tokens), 0) AS cache_creation,
  COALESCE(SUM(m.cache_read_tokens), 0)     AS cache_read
FROM messages m
JOIN sessions s USING (session_id)
WHERE m.role = 'assistant'
GROUP BY 1, 2, 3, 4;

CREATE UNIQUE INDEX usage_daily_unique ON usage_daily (day, host, project_path, model);
```

`DEFAULT 'local'` backfills every existing row atomically — no separate UPDATE migration needed for the ~519 sessions and ~300k events already in the DB. On Postgres 17, `ADD COLUMN ... DEFAULT 'literal'` is metadata-only since v11; the table is not rewritten.

`model_pricing` is unchanged — pricing is global. `_ingest_cursors` is unchanged — it keys on full `source_file` paths, and remote files live at `<repo>/.cca/remotes/<host>/.claude/...` so they are naturally distinct from local paths. No collision possible.

**Why denormalize host to all 8 tables and not just `events`?** Per-host token sums and per-host cost queries hit `sessions`/`messages`/`tool_calls` directly. Joining back through `events` for every aggregate would be a measurable hit on `/hosts` and the home-page token headline at the row counts already in the DB (~85k tool_calls, ~221k messages). The cost is 6 extra columns; the benefit is single-table aggregates everywhere — and any chart that already uses `usage_daily` (the dashboard's stacked-area-by-day) can layer the host filter on cheaply via the new `host` column in the view.

### 4.2 New table: `host_sync_state`

```sql
CREATE TABLE host_sync_state (
  host                    TEXT PRIMARY KEY,
  last_pulled_at          TIMESTAMPTZ,
  last_had_data_at        TIMESTAMPTZ,
  current_interval_hours  INTEGER NOT NULL DEFAULT 3,
  consecutive_empty_pulls INTEGER NOT NULL DEFAULT 0,
  last_error              TEXT,
  last_error_at           TIMESTAMPTZ,
  consecutive_errors      INTEGER NOT NULL DEFAULT 0
);
```

Drizzle schema file: `packages/db/src/schema/hostSyncState.ts`, re-exported from `packages/db/src/schema/index.ts`.

`last_error_at` is tracked separately from `last_pulled_at` so a transient SSH failure does not get counted as "empty data" and trip the backoff. Backoff only advances on a successful-but-empty pull.

### 4.3 Writer wiring

`apps/ingester/src/writer/` and `apps/ingester/src/backfill/` currently have no host concept. The full list of files to change is broader than a "few surgical edits" — it is **9 source files** and **5 test files**. None of the changes are individually large, but there is no single chokepoint to thread host through.

**Source files:**

| File | Change |
|---|---|

| `packages/db/src/schema/sessions.ts` | Same. |
| `packages/db/src/schema/messages.ts` | Same. |
| `packages/db/src/schema/toolCalls.ts` | Same. |
| `packages/db/src/schema/ancillary.ts` | Same on `prompts_history`, `file_snapshots`, `shell_snapshots`, `todos`. |
| `packages/db/src/schema/hostSyncState.ts` | New table (§4.2). Re-export from `index.ts`. |
| `apps/ingester/src/writer/events.ts` | `insertEventsBatch(events, { host })` — stamp on every row. |
| `apps/ingester/src/writer/deriveMessages.ts` | Add `host` to the row map. |
| `apps/ingester/src/writer/deriveToolCalls.ts` | Add `host` to the row map. |
| `apps/ingester/src/writer/deriveSessions.ts` | See §4.3.1 below — non-trivial SQL change in the rollup query. |
| `apps/ingester/src/backfill/orchestrator.ts` | `backfillAll(claudeHome, opts)` accepts `host: string` (default `'local'`). Threads through `ingestTranscriptFile` and the four ancillary calls. |
| `apps/ingester/src/backfill/ancillary.ts` | All four functions (`ingestHistory`, `ingestTodos`, `ingestFileHistory`, `ingestShellSnapshots`) take a `host` parameter and stamp it on their rows. |
| `apps/ingester/src/daemon/liveIngest.ts` | One new arg, hard-wired to `'local'` at the call site. The daemon never writes any other host. |
| `apps/ingester/src/cli.ts` | Existing `backfill` and `daemon` actions pass `host: 'local'`. New `sync` action passes `host: '<name>'`. |

**Test files:**

| File | Change |
|---|---|
| `apps/ingester/src/writer/*.test.ts` (4 files) | Existing test fixtures update to include `host` in insert/expect blocks. |
| `apps/ingester/test/integration.test.ts` (or current equivalent) | Add a backfill-with-host case verifying every table receives the right value. |

**Default host.** Every new code path takes `host` as a required parameter — no implicit defaults at the writer layer. Defaults are only set at the orchestrator boundary (`backfillAll`'s `opts.host = opts.host ?? 'local'`) so callers cannot silently bury remote data under `'local'` by forgetting an arg.

#### 4.3.1 `rollupSessions` host derivation

`apps/ingester/src/writer/deriveSessions.ts` runs an `INSERT ... ON CONFLICT (session_id) DO UPDATE` whose SELECT groups by `session_id` and uses `array_agg(...)[1]` for project_path / git_branch / cc_version. To add `host`, two changes:

1. The SELECT gains `MIN(e.host) AS host`. Since every event for a given session shares one host (no cross-host session-id collision is possible), `MIN` is a stable single-value pick that avoids the cost of `array_agg`.
2. The INSERT column list adds `host`, the VALUES gets `host`, and the EXCLUDED clause adds `host = EXCLUDED.host` so re-rollups stamp the same value (idempotent).

The existing `CASE WHEN sessions.status = 'active' THEN ... ELSE EXCLUDED.status END` clause in the same INSERT is unaffected — host and status are independent.

## 5. Configuration: `cca.remotes.json`

A JSON file at the repo root, gitignored (added to `.gitignore`):

```json
[
  { "host": "hostinger", "ssh": "ssh_hostinger", "claudeHome": "~/.claude" },
  { "host": "picoclaw",  "ssh": "ssh_picoclaw",  "claudeHome": "~/.claude" }
]
```

Fields:

| Field | Purpose |
|---|---|
| `host` | Label stamped on every ingested row; shown in the UI; primary key in `host_sync_state`. |
| `ssh` | SSH alias (resolves via `~/.ssh/config`). |
| `claudeHome` | Remote Claude Code data dir; tilde-expanded by `ssh` itself. Defaults to `~/.claude` if omitted. |

Validated by a small Zod parser on every `cca sync` invocation. Errors are surfaced as a non-zero exit and a clear message; a malformed file does not corrupt `host_sync_state`.

**Validation rules** (all enforced before any rsync runs):

| Rule | Reason |
|---|---|
| `host` must match `/^[a-z0-9][a-z0-9_-]*$/` | `host` is interpolated into a filesystem path (`<repo>/.cca/remotes/<host>/`). A value like `..` or `foo/bar` would be a path-traversal bug; a value with shell metacharacters would be unsafe even when args are passed as an array. |
| `host` must not equal `'local'` | That label is owned by the live daemon path; allowing it would cause the sync runner to overwrite local data. |
| `host` values must be unique across the registry | Two configs colliding on the same key produce undefined ingest behavior. |
| `ssh` must be a non-empty string | Required by rsync. |
| `claudeHome` defaults to `'~/.claude'` if omitted; must not be empty | Tilde expansion happens on the remote, not locally. |

**Config vs. state separation.** `cca.remotes.json` holds *what the user wants to sync*; `host_sync_state` holds *where the system is*. Hand-edits to the registry never corrupt backoff state, and a `cca sync --reset-state <host>` (deletes the state row) never loses the host list.

## 6. Sync state machine

### 6.1 Backoff schedule

Per-host interval ramp:

```
3h  ──(empty pull)──►  6h  ──(empty pull)──►  12h  ──(empty pull)──►  12h  ...
                                                       │
                                                       ▼
                                          (any non-empty pull resets to 3h)
```

`consecutive_empty_pulls` is the source of truth; `current_interval_hours` is derived from it on update:

| `consecutive_empty_pulls` | `current_interval_hours` |
|---|---|
| 0 | 3 |
| 1 | 6 |
| ≥ 2 | 12 |

A non-empty pull resets `consecutive_empty_pulls = 0` and updates `last_had_data_at = now()`.

### 6.2 Triggering

A single launchd plist `infra/launchd/com.aporb.cca.sync.plist` runs `pnpm --filter @cca/ingester exec tsx src/cli.ts sync` every **3 hours** (`StartInterval = 10800`). The launchd job always wakes at the minimum interval; the sync command itself decides per-host whether each host is due, by comparing `last_pulled_at + current_interval_hours` to `now()`. This avoids fighting launchd's static-interval scheduling model and keeps all backoff state in Postgres.

The same launchd FDA caveat as the existing daemon plist applies. The fallback is the same: invoke `cca sync` from a terminal manually, or grant FDA to the launchd executor. Both are documented in `README.md` already; this spec extends that documentation, doesn't change the constraint.

### 6.3 Per-host run loop

On each `cca sync` invocation, for each host in `cca.remotes.json` (sequentially, not in parallel — keeps logs readable and avoids hammering an SSH server with simultaneous handshakes from one client):

1. Read or insert the `host_sync_state` row (initial defaults: `current_interval_hours=3`, `consecutive_empty_pulls=0`, `consecutive_errors=0`, all timestamps NULL).
2. **Due check:** if `last_pulled_at IS NOT NULL AND last_pulled_at + (current_interval_hours * INTERVAL '1 hour') > now()`, skip with a one-line log. (When `last_pulled_at IS NULL`, the host has never been pulled — always run.)
3. **Rsync:** `rsync -az --delete-after --info=stats2 ssh_<host>:<claudeHome>/ <repo>/.cca/remotes/<host>/.claude/`. Capture stdout; parse `"Number of regular files transferred: <n>"` and `"Total transferred file size: <n> bytes"`. **Rsync version note:** macOS ships rsync 2.6.9, whose `--info=stats2` output differs from rsync 3.x. The implementation runs `rsync --version` once at startup, branches on `>=3.0`, and falls back to `--itemize-changes` + a non-zero-line count for older versions. If neither parse succeeds, treat the outcome as "non-empty success" (conservative — we'd rather over-ingest than skip).
4. **Outcome classification:**
   - **Error** — non-zero exit code: set `last_error`, `last_error_at = now()`, `consecutive_errors += 1`. Do NOT touch `consecutive_empty_pulls` or `current_interval_hours`. Skip ingest. Continue to next host.
   - **Empty success** — exit 0, files transferred = 0: `consecutive_empty_pulls += 1`, `consecutive_errors = 0`, `last_error = NULL`, set `last_pulled_at = now()`. Skip ingest. Continue.
   - **Non-empty success** — exit 0, files transferred > 0: `consecutive_empty_pulls = 0`, `consecutive_errors = 0`, `last_error = NULL`, set `last_pulled_at = now()`, `last_had_data_at = now()`. Run ingest (§7). Continue.
5. Log a one-line summary per host to `~/Library/Logs/cca/sync.log`.

The `--force` flag on `cca sync` overrides the due check in step 2 (still runs the rest unchanged).

### 6.4 Failure isolation, locking, and crash semantics

**Failure isolation.** A failure on host A (SSH unreachable, rsync error, ingest crash) does not abort the run. The run continues to host B. Each host's state row is updated independently. After **3 consecutive errors** on a host, the web UI surfaces a banner (§8.4). Threshold rationale: at the 3h scheduled cadence, 3 errors signal at least 9 hours of failed pulls — roughly the lower bound at which "transient" feels wrong. Thresholds higher than 3 push the alert past the practical attention window.

**Per-host file lock.** A `flock(2)` on `<repo>/.cca/remotes/<host>/.lock` is acquired around each per-host run (rsync + ingest + state-row update). This prevents a manual `cca sync --force` from racing the scheduled run on the same host — concurrent rsyncs against the same destination dir with `--delete-after` can produce surprising results (one process deletes a file the other just wrote). The lock is per-host, not global, so two different hosts still run sequentially under the same `cca sync` invocation but a manual run can interleave with a scheduled run on a *different* host without contention. (We do not use a Postgres advisory lock — the resource being protected is the local filesystem, not the DB.)

**Crash mid-run is by design safe.** If `cca sync` is killed between rsync completion and ingest completion: `last_pulled_at` and `consecutive_empty_pulls` are NOT yet updated (they're written only in step 4 of §6.3, after outcome classification), so the next scheduled run treats the host as still due and re-runs. `_ingest_cursors` may have advanced for files that ingested before the crash; the next run resumes from those offsets. The mirror dir is consistent because rsync is atomic per-file. Net effect: a mid-run crash costs one extra scheduled cycle to recover, with no data loss and no duplicate inserts.

**Concurrency with the local daemon.** The daemon and a sync run can write to Postgres simultaneously without conflict. Postgres handles row-level concurrency; the only resources at issue are: (a) `events.uuid` (PK) — different transcript files on different machines produce different UUIDs from the parser, so no collision is possible; (b) `_ingest_cursors` (PK on `source_file`) — local and remote files have distinct paths.

## 7. Ingest path

### 7.1 Directory layout

```
<repo>/.cca/remotes/
├── hostinger/
│   └── .claude/
│       ├── projects/
│       ├── history.jsonl
│       ├── todos/
│       └── ...
└── picoclaw/
    └── .claude/
        └── ...
```

`/.cca/` is added to `.gitignore`. Adopting `.cca/` rather than a sibling directory keeps everything inside the repo so backups and developer cleanup are obvious.

### 7.2 Reuse of the backfill orchestrator

`backfillAll(claudeHome, opts)` in `apps/ingester/src/backfill/orchestrator.ts` already accepts a path. The change is to thread a `host` field through `opts` to the writers:

```ts
backfillAll('<repo>/.cca/remotes/hostinger/.claude', { concurrency: 6, host: 'hostinger' })
```

The existing local backfill call site becomes `host: 'local'`. The daemon's tailer remains hard-wired to `host: 'local'` because it only watches `$CLAUDE_HOME`.

`_ingest_cursors` works without modification: every cursor row is keyed by absolute file path, and remote-mirror paths are distinct from local paths. Re-running rsync + ingest is idempotent — already-ingested bytes are skipped via the existing offset cursor.

### 7.3 First-run cost

The first sync per host backfills the entirety of that remote's `~/.claude`. Expected magnitude is the same order as the local Mac (which today produces ~300k events / ~2.8 GB DB). One-time. Subsequent syncs only ingest the bytes rsync delivers, so they are incremental.

### 8.3 New `/hosts` page

Card per host. Cards laid out as a responsive grid (3-up desktop, 1-up mobile). Each card:

- Host label (chip with hash-derived color).
- **Last sync:** relative time + absolute on hover.
- **Last active session:** relative time on the actual `MAX(sessions.started_at) WHERE host = X`. **This can lag `last_pulled_at` by hours-to-days when a remote was idle through a backoff window** — the card layout makes both timestamps visible side-by-side so the user can tell "no remote activity" from "sync is broken".
- **Sessions:** count over global window.
- **Tokens:** stacked bar of in / out / cache (creation + read), with totals.
- **Cost:** USD over window.
- **Top model:** name + % of cost.
- **Sync health:** ●green / ●yellow (1–2 consecutive errors) / ●red (≥3 consecutive errors, matches §6.4 threshold). Clicking expands the most recent error.

Cards sortable by tokens / cost / sessions / last active. Clicking a card navigates to `/sessions?host=<host>` pre-filtered.

Backed by a new `apps/web/lib/queries/hosts.ts` module, following the per-route query module pattern established in the dashboard redesign. Tested with a vitest test against the real DB, like the other `lib/queries/*` tests.

## 10. Error handling

| Failure mode | Handling |
|---|---|
| `ssh` not reachable | Captured as rsync non-zero exit. `last_error` set. `consecutive_errors` incremented. Backoff state untouched. |
| Remote `claudeHome` missing | Same as above — rsync fails, error row recorded. |
| Corrupted JSONL on remote | Existing parser-level skip + log path (the ` `-in-string-values issue from the existing system). Ingest continues for other files. |
| `cca.remotes.json` malformed | Sync command exits non-zero with a clear message. No rows touched. No rsync runs. No mirror dirs created. |
| Migration on populated DB | `ALTER TABLE ADD COLUMN ... DEFAULT 'local'` is metadata-only on PG 11+. Single transaction. Run during a quiet moment via `pnpm db:migrate`. The `usage_daily` rebuild is a single SELECT and finishes in seconds at current row counts. |
| Disk full at `<repo>/.cca/remotes/` | Rsync fails. Surfaced as sync error. Operator clears space and retries. |
| Two `cca sync` runs racing on the same host | Prevented by the per-host `flock` in §6.4. The second runner blocks on the lock until the first finishes, then re-checks the due-state and likely skips. |
| Sync run killed mid-ingest | By design safe — see §6.4. State row is not advanced until outcome classification completes; next scheduled run resumes via `_ingest_cursors`. No duplicate inserts; at most one extra cycle to recover. |
| User removes a host from `cca.remotes.json` | Sync runner stops pulling that host. Existing `host_sync_state` row, `events.host = '<removed>'` rows, and the host's mirror dir under `<repo>/.cca/remotes/` are all left in place — historical data stays queryable in the UI under the old label. To fully remove: `cca sync --reset-state <host>` (deletes the state row), then `psql DELETE FROM events WHERE host = '<host>'` (and the same on the seven other event-derived tables), then `rm -rf <repo>/.cca/remotes/<host>`. Deliberately manual; we don't ship a one-button purge. |
| User renames a host | Treated as "remove old + add new". The old label's rows become orphans until a manual cleanup. The mirror dir under the old label remains. Operator runs the removal sequence above on the old name before pulling under the new one. |

## 13. Open questions / deferred

- **Per-host `claudeHome` overrides** other than `~/.claude`. Supported in the schema; not exercised on day one.
- **Cross-host cost split charts on `/`** (stacked area split by host instead of model). Easy follow-up; not in v1.
- **Pruning `<repo>/.cca/remotes/` after ingest.** Currently we keep the rsync mirror so reruns are incremental. If disk pressure becomes a concern, prune older files via `--remove-source-files` or a separate cleanup job.
- **Sync from a remote that *also* runs cca and writes to its own DB.** Out of scope; assumes remotes only run Claude Code, not the cca daemon.
- **Stale ingest cursor after remote-side log rotation.** Not handled here; same hazard the local pipeline already inherits.
- **One-button "remove host" purge** that combines the `--reset-state` + `DELETE FROM` + `rm -rf` sequence in §10. Deliberately deferred; the manual sequence is rare enough that automating it isn't worth the footgun.
